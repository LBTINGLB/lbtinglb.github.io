An Even Easier Introduction to CUDA

  • __global__: a specifier added to a function which tells the CUDA C++ compiler that this is a function that runs on the GPU and canbe called from CPU code. It’s known ad kernels and code that runs on the GPU is often called device code while code that runs on the CPU is host code.
  • Memory allocation in CUDA:
    • Unified memory provides a single memory space accessible by all GPUs and CPUs in the system. cudaMallocManaged() + cudaFree()
    • cudaDeviceSynchronize(): let CPU wait untill the kernel is done before it accesses the results(because CUDA kernel launches don’t block the calling CPU thread)
    • nvprof: a profile tool provided by Nvidia
    • <<<numBlocks, blockSize>>>: execution configuration. It tells the CUDA runtime how many parallel threads to use for the launch on the GPU.
    • CUDA GPUs have many parallel processors grouped into Streaming Multiprocessors, or SMs. Each SM can run multiple concurrent thread blocks. As an example, a Tesla P100 GPU based on the Pascal GPU Architecture has 56 SMs, each capable of supporting up to 2048 active threads. To take full advantage of all these threads, I should launch the kernel with multiple thread blocks.
    • numberOfSMs -> number of thread blocks in the grid(gridDim.x) -> number of threads in each block(blockDim.x)
    • blockId.x: the index of the current thread block in the grid
    • threadIdx.x: the index of the current thread within the block

Unified Memory for CUDA Beginners

  • Unified memory: a single memory address space accessible from any processor in a system. This hardware/sfotware technology allows applications to allocate data that can be read or written from code running on either CPUs or GPUs. Pascal GPU architecture is the first with hardware support for virtual memory page faulting and migration via the Page Migration Engine.
  • Page fault: when a running program is accessing a memory page that is not currently mapped by the memory management unit into the virtual address space of a process.
  • The Page Migration engine allows GPU thread to fault on non-resident memory accesses so the system migrate pages on demand from anywhere in the system to the GPU’s memory for efficient processing.

Maximizing Unified Memory Performance in CUDA

  • what happens when a cudaMallocManaged allocation is accessed on GPU:
    • alloate new pages on the GPU
    • unmap old pages on the GPU
    • Copy data from the CPU to the GPU
    • Map new pages on the GPU
    • Free old CPU pages
  • page faulting is a significant overhead to streaming performance of unified Memory on current generateion GPU architectures.
  • density prefetching: works by testing how much of the predefined region has been or is being transferred; if it meets a certain threshold the driver prefetches the rest of the pages
  • warp-per-page approach: Instead of having multiple hardware warps accessing the same page, we can divide pages between warps to have a one-to-one mapping and have each warp perform multiple iterations over the 64k region.
  • Overlapping kernels and Prefetches: on-demand access with migration will never beat explicit bulk data copies or prefetches in terms of performance for large contiguous memory regions.It’s the price for simplicity and ease of use.
  • It’s important to make sure that we have the device-to-host prefetch issued in a busy stream while the host-to-device prefetch is issued in an idle stream so that we have overlap to optimize.
  • unified memory programing model
    • All managed memory must be allocated from the host or at global scope (see the next section). Allocations on the device heap using malloc() in a kernel will not be created in the managed memory space, and so will not be accessible to CPU code.

Grid-Stride loops

  • Common CUDA guidance is to launch one thread per data element

  • Benefit:

    + All addressing within warps is unit-stride, so we get maximum memory coalescing.
+ Scalability and thread reuse
+ Debugging
+ Portability and readability
