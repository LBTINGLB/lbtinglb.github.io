Thread Management in Mac

  • The link points to some archive doc. The new Apple development doc is here.

Thread Management

  • Every application starts with a single thread, which runs the application’s main function. Application can spawn additional threads, each of which executes the code of a specific function.
  • Because they are inside the same process space, however, all threads in a single application share the same virtual memory space and have the same access rights as the process itself.

Thread Costs

  • Each thread requires the allocation of memory in both the kernel memory space and your problem’s memory space. Launching a thread can be relatively expensive because of the required interactions with the kernel. It needs 90microseconds to create a thread(the operation objects can often create threads more quickly. It would use pool or threads already residing in the kernel to save on allocation time).

Creating a Thread

  • In all cases, you must have a function or method to act as your thread’s main entry point and you must use one of the available thread routines to start your thread.
  • NSThread: a detached thread(its resources are automatically reclaimed by the system when the thread exits)

Using POSIX Threads

  • Cocoa(API)): Cocoa is Apple’s native object-oriented application programming interface (API) for their operating system macOS.
  • POSIX: Portable Operating System Interface, a family of standards specified by IEEE for maintaining compability between different OS
  • Detached vs joinable POSIX threads: When a detached thread terminates, its resources can be released immediately istead of having to wait for the thread to be joined before the resources can be released
  • communication between threads: ports, conditoins or shared memory. For long-lived threads, you should almost always set up some sort of inter-thread communications mechanism to check and shut thread down
