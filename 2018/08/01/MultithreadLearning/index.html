Writing Multithreaded Applications in C++

  • Why use threads: manage multiple points of interaction, alternate method for nonblocking operations, hiding memory latency
  • thread is not CopyConstructible or CopyAssignable. But it’s MoveConstructible and MoveAssignable
  • thread, future, promise, mutex and atomic
    • how to launch a thread: thread.join
      • how to use promise and future: create a promise-> thenfuture=promise.get_future()->future.wait()// wait for result->thread.join()`
  • producer-consumer scenario:
  • std::mutex: a synchronization primitive that can be used to protect shared data from being simultaneously accessed by multiple threads(neither copyable nor movable)
  • std::lock_guard: The lock_guard object locks a mutex while in scope. After the object is out of scope, the mutex lock is released.
  • Two Tasks. one process. How to do asynchronous computation
    + `std::promise`: provide a value for a future. Computation is performed independently from the promise object and the result is simply passed through the object to the future using the set_value() method.
+ `std::packaged_task`: The second most flexible way to provide a value for a future. The constructor takes a function and uses the return value of that function to set the value for the future.
    • std::async: A high-level utility method to provide a value for a future providing the least flexibility, but also the simplest way to execute an asynchronous computation.
  • When should I use which?
    + `std::async` automatically begins execution upon calling it. The caller has no control over it.

Others

  • When to use promise over async or packaged_task

    • std::async:

      • not always starts a new thread, passing std::launch::async as a first parameter to force it.
      • std::~future destructor can block until new thread finishes
      • if the std::future is stored in a temporary-life object, std::async call will block on spot, so the following block will take 10 seconds if you remove the auto f = initializations

    • std::packaged_task: it has nothing to do with threads, it’s just a functor and a related future

      • Instead of just a function you can initialize std::thread with a packaged_task. Consider the following - packaged_task is a really nice way of getting future:
      • not copyable, so you move it to a new task with std::move
    • std::promise: you can pass a value to new thread without need of any additional synchronizing mechanism.

  • use std::async only for simple things, e.g. to make some call non-blocking, but bear in mind the comments on blocking above.

  • use std::packaged_task to easily get future, and run it as a separate thread

std::thread{ std::move(package), param }.detach(); or std::thread t { std::move(package), param };

  • use std::promise when you need more control over the future.

Thread local storage

+ a CS method that uses static oro global memory local to a thread. TLS is used in some places where ordinary, single-threaded programs would use global variables, but where this would be inappropriate in multithreaded cases.

thread_local in C++11

  • each thread has its own copy of the thread_local variable and they are entirely independent of each other.
  • even it’s thread_local, you can pass the address of the thread_local variable to some thread so that the value can be assigned to.
  • thread_local variables are initialized “before first use”
