Chapter1: Deducting Types

  • Universal reference: They exists and are not the same as lvalue reference or rvalue reference

Item 1: Understanding type deduction

  • During tempalte type deduction, arguments that are references are treated as non-references.

  • When deducing types for universial reference parameters, lvalue arguments get special treatments - both T and ParamType are deduced to be lvalue references.

  • During template type deduction, arguments that are array or function names decay to pointers, unclass they’re used to initialize references.

Item 2: Understanding auto deduction

  • 3 cases:
    • pointer or reference, but not a universal reference(const auto& a = x; )
    • a universal reference (auto&& uref1 = x;)
    • neither a pointer or reference (auto x = 27;)
  • special type deduction which is the only difference between auto and std::initializer_list
    • auto x3 = {27}; // type is std::initializer_list

      Item 3: Understand decltype

  • std::vector is special. Operator [] does not return a bool&. Instead, it returns a brand new object. So a decltype is needed here.

  • For lvalue expressions of type T other than names, decltype always reports a type of T&. Ex decltype(x) - > int but decpltype((x))-> int& where x = 4;

  • c++14 has support for decltype(auto) that it perfomrs the type deduction using the decltype rules

Item 4: Know how to view deduced types

  • When? editing the code -> IDE; during compilation(P32); at runtime(typeid().name())
  • typeid().name() is not reliable because it’s treating type as if it has been passed to a tempalte function as a by-value parameter
  • Boost.TypeIndex is your friend

Item: 14: Declare functions noexcept if they won’t emit exceptions

  • in C++11, a function might emit an exception or it guaranteed that it wouldn’t
  • it permits the compiler to generate better object code
  • example: std::vector::push_back would move if it can and copy if it must
  • in C++11, all memory deallocation functions and destructors-both user-defined and compiler-generated-are implicitly noexcept
  • noexcept is particularly valuable for the move operations, swap, memory deallocation functions and destructors

Item 15: Use constexpr whenever possible

  • The value of constexpr is known at translation time.
  • When a constexpr function is called with one or more values that are not known during compilation tiem, it acts like a normal function.
  • constexpr is part of an object’s or function’s interface which means if you remove it, you may cause arbitrarily large amounds of client code to stop compiling

Item 16: Make const member functions thread safe

  • use mutable keyword when you want to change something via a const function or in a class
  • std::atomic: different threads can simultaneously operate on the instance, without raising undefined behavior. Good for only a single variable or memory location

  • std::memory_order:

    • memory_order_release: no reads or writes in the current thread can be reordered after this store. All writes in the current thread are visible
    • memory_order_acquire: a load option with this memory order performs the acquire operation on the affected memory location: no reads or writes in the current thread can be reordered before this load. All writes in other threads that release the same atomic variable are visible in the current thread.

  • extended reading: release-consume ordering
