Chapter 14. Contributing Code

14.1 Coding Considerations

  • Variables, methods, class names: changing capitalization
  • instance variables and methods: begin with a capital letter
  • local variable: begin with a lower-case letter
  • Class name: previx with vtk
  • instance variable: make them protected or privated. Access its instance variables is through Set/Get methods (check VTK/Common/vtkSetGet.h)
  • PIMPL
    • Pointer to IMPLementation
      • data hiding used to hide internal implementation details and data from the user of the library,
      • remove compilation dependencies and decrease recompilation time
      • binary compatibility Since the class declaration doesn’t change, it is safe to just update the library (assuming you are creating a library)
    • definition
      • d-pointer, an opaque pointer defined in C++, used to hide the implementation details of an interface from ordinary clients, so that the implementation may be c hanged without the need to recompile the modules using it.

14.4 Using STL

  • it can be problematic in cross-platform, causing namespace management, DLL bvoundary problems and threading issues.
    • all STL references should be contained in a .cxx calss implementation file. Neverin the .h header file.

14.5 Manging Include Files

  • In the build process, whenever any one of the header files changes, all the code that depends on that file must be recompiled.
    • so, a classs definition(.h) file should include at most one other #include file-its superclass definition file. All other #includes’s should be placed in the class implementation(.cxx) file.

Chapter 15. Managing Pipeline Execution

  • Filter
    • algorithm: processing information and data
    • executive object: tell an algorithm when to execute and what information and data to proces
    • input connection: stored in Input ports, which reference the output ports of other filters providing information and data

15.1 Information Objects

  • Pipeline Information
    • one pipeline information object per output port
  • Port Information
    • objects hold information about the data types produced on output ports and consumed by input ports
  • Request Information
    • objects hold information about a specific request being sent to an executive or algorithm
  • Data Information
  • Algorithm Information

15.2 Pipeline Eecution Models

  • execution model: a set of requests defined by a specific executive
  • vtkAlgorithm::ProcessRequest needs input pipeline info and output pipeline info

15.3 Pipeline Information flow

  • downstream flow: modify the output pipeline information
  • upstream flow: modify the in pu t pipeline information

15.4 Interface of information Objects

  • vtkInformation: provide a heterogeneous key-to-value map.

    • type of a key object: interpret the key values
    • Ex: create a vtkInformation instance - > create a key -> Set the value

Section 1: VTK Data Structures

  • vtkDataObject
    • definition: represents a “blob” of data
    • contain instance of vtkFieldData (a set of arrays)
    • no geometric/topological structure
    • superclass of all VTK data objects
  • vtkDataSet (subclass of vtkDataObject)
    • definition: a data object with geometric/topological structure
    • geometry(points) + topology(cells)
      • pointData = aDataSet->GetPointData()
      • cellData = aDataSet->GetCellData()
    • associated point- and cell-centered data which is stored in vtkDataArray objects and managed by vtkPointData and vtkCellData
  • vtkDataArrays
    • definition: pointer to contiguous buffer
    • of ints, floats, chars, strings, variants
    • array = type [tuples][components]
    • methods:
      • Set: you just write stuff without range check. So use SetNumberOfTuples() prior to invokeSet` method
      • Insert: you do range check, allocate memory as necessary, write into index specified”
        Common Operation: Copy Input to Output
  • In a filter we wish to copy the input to the output(check detail on pdf: P20)
    • PassData(), CopyAllocate(), CopyData()
  • vtkFieldData VS vtkDataSetAttribute
    • vtkFieldData: add/remove named arrays, a model of a field as m x n matrix of data values
    • vtkDataSetAttribute: represent and manipulate attribute data(e.g, scalars, vectors, normals, texture coordinates, tensors, global ids. pedigreeids, and field data)
      Exercise 1: Attribute Data
  • Dataset API
    • all simple dataset types are subclasses of vtkDataSet, therefore all data can be treated as instances of vtkDataSet
      Exercise 2 - Creating vtkPolyData

    • Observer pattern is used when there is one-to-many relationship between objects such as if one object is modified, its depenedent objects are to be notified automatically. Observer pattern falls under behavioral pattern category.

    • Command pattern is a data driven design pattern and falls under behavioral pattern category. A request is wrapped under an object as command and passed to invoker object.   • MTime in VTK:
    • When graphic card is rendering a scene, it’s too expensive to update every geometry. So we use MTime to log when the geomery is updated, then for every time step we only update the things that has been changed.

  • actors are tied to a particular render window
    • Essentially yes, mainly because an actor has a mapper and mappers should not be shared between windows.
  • mappers are likewise tied to a particular render window
    • Yes. Mappers create buffer objects that are tied to a window’s OpenGL context. Right now we do not support sharing buffers between contexts (it could be added) so having the same mapper in two windows would either cause the buffers to be rebuilt and sent down every frame or an OpenGL error. You could share an actor/mapper between two windows if you knew only one would active at a time. You would need to call release graphics resources every time you switch which window is active to make sure the buffers etc all rebuilt. But the general answer is they are not designed to be shared.

VTK New Pipeline

  • downstream request: input to output
  • ProcessRequest: request - specifies what you are asking the algorithm to do, inputVector - inputs to this algorithm and outputVector - one for the outputs of this algorithm
  • Typical Pipeline Execution: Request Information(What output would look like?) -> Request Update Extent(an algorithm should take the update extents in its output’s information and then set the correct update extents in its input’s information) -> Request Data(fill in the output ports)
  • streaming in VTK

Typical Pipeline Execution

  • First the use instantiates an Algorithm such as vtkImageGradient, when the algorithm is instantiated it will automatically create a default executive and connect the algorithm and executive together. The algorithm will also call FillInputPortInformation and FillOutputPortInformation on each input and output port respectively.
  • request information: It askes the algorithm to provide as much information as it can about what the output data will look like once the algorithm has generate it.
  • Request Update Extent
  • Request data
  • BUG: vtkStreamingDemandDrivenPipeline (0x22db870): The update extent specified

vtkNew vs vtkSmartPointer

  • vtkSmartPointer is the older one. It’s more suitable for the case where the class has API to set what the member variable is pointing to, while maintaining ownership of that class. Ownership of the instance is maintained. The class may or may not instantiate the initial object
    and the instance being pointed to can be changed.
  • vtkNew has no assignment operator or copy constructor. It has a strong ownership, where the class instantiates an object which it owns for the lifetime of the class.
  • vtkWeakPointer: Weak ownership, where the class does not instantiate an object. If the object instance is deleted, then the weak pointer is null, which avoids dangling pointers without ownership.
