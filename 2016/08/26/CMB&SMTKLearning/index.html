CMB official website

  • CMB - Computational Model Builder - leverage several powerful open-source tools and integrate them into an application framework (pitstatus)
  • Two libraries for CMB:
    • Simulation MOdeling Toolkit(SMTK):
      • Geometric domain + information associated with the geometry(BC or material properties)
    • Remote Meshing/Modeling Utilities(ReMUs)
      • provides remote functionality to different modeling kernels and meshing systems, which allows users to access functionality that is not supported on their users’ platforms or to make use of HPC systems, which can better provide the computational resources.

CMB Documentation

  • What’s CMB?
    • Computational Model Builder (CMB) is a suite of applications that provide a customizable workflow for the entire lifetime of a computational simulation.
  • workflow
    • specifying the governing equations of state of the system to be simulated, their free parameters, and other quantities required by the simulation;
    • importing, constructing, or fitting a geometric model of the physical domain of the simulation;
    • discretizing the geometric model into a mesh suitable for approximating functions which enable people to make a decision based on the outcome of the simulation;
    • associating material properties, boundary and initial conditions, and other attributes to regions of the mesh.
      queueing the simulation (or an ensemble of simulations) for execution;
      monitoring simulation progress and inspecting results after the simulation has run;
    • integrating simulation results, experimental measurements, and other knowledge in order to make a decision about how to proceed.

SMTK’s Attribute System

  • provides a way to model and represent non-geometric simulation data such as material properties, boundary conditions, and solver parameters.
  • Key Concepts:
    • Attribute: instances represent a dictionary of named values map[class] = subclasses
    • Definition: instances hold the set of possible key-value pairs that must be present in Attribute instances that reference them. (Ex. deflection, temperature, and voltage boundary condition definitions might all inherit a Dirichlet boundary condition definition)
    • Item: istances hold values in an attribute key-value pair (type of storage)
    • ItemDefinition: instances constrain the number of values that an item istance may contain (Ex. Only a single value may be held in the item and it’s a floating point value)
    • System: instances hold collections of attributes associated with a particular purpose such as defining a simulation’s input deck
  • Components of the attribute system
    • XML file: specify the unique inputs for a simulation code, including, for example, data types, valid ranges, and default values.
  • Conditional items:
    • One particular workflow SMTK supports is attributes that can be defined by multiple, distinct sets of values. For example, there are many ways to define a circle
  • Template File Syntax:
    • : 1 or 2
    • Category Section
      • ,
    • Analysis Section
    • Definitions Section

SMTK’s Geometric Model System

  • a system allows you to interact with multiple modeling sessions that may each interact with different solid modeling kernels
  • Manager - instances contain model topology and geometry, key-value store of faces, edges, vertices
  • Session - instances relate entries in a model Manager to a solid modeling kernel
  • Operator - modeling operations that a modeling kernel provides for marking up, modifying, or even generating modeling entities from scratch
  • EntityRef - instances are lightweight references into a model Manager’s storage, referencing to its related entities, properties, tessellation data
  • DescriptivePhrase - instances provide a uniform way to present model entities and the information associated with those entities to users
  • SubphraseGenerator - instances accept a DescriptivePhrase instance and enumerate its children
  • Sessions
      1. transcribe modeling-kernel entities into SMTK’s storage
      1. keep a list of Operators that can be used to modify the model
    • Session:CGM
      • bridge Common Geometry Model(CGM)meta-modeling kernel
    • Session:Discrete
      • bridge a VTK-based discrete modeling kernel, accessing 2-D and 3-D polygonal and polyhedral mdoels
    • Session: Exodus
      • it’s not technically a modeling kernel but does allow you to mark up meshes saved in the Exodus-II file format for use in an existing simulation
    • Session: Polygon
      • bridge Boost.polygon’s modeling kernel
        • why floating point coordinates are wasteful
        • The increasing-resolution-with-closeness-to-zero characteristic of floating point coordinate numbers exceeds the needs of 3D objects since a single resolution for all of a given object’s coordinates is quite adequate.

SMTK’s Mesh System

  • Key Concepts are based on MOAB
    • Handle - instance refer to a single mesh entity, which may be an primitive mesh element (a Cell, e.g., a triangle or quadrilateral) or a set of mesh elements (a MeshSet) grouped together for some purpose.
    • HandleRange - instances refer to a range of mesh entities.
    • MeshSet - instances group HandleRange instances toether so that they can be marked with properties such as a common association with a model entity, material, boundary condition, or initial condition.
    • Collection - instances hold related MeshSets together.
    • Manager - instances contain mesh collections and provide an interface to an underlying mesh package, such as MOAB that implements methods to access the mesh.
  • IO:
    • WriteMesh - MOBA(h5m, mhdf), Exodus II
    • ImportMesh - same

    Implementation an STMK operator

  • operator consist of three componets:
    • a class that implements the action of the operator, as a subclass of smtk::model::Operator.
    • an SMTK attribute definition that describes the parameters the operator accepts (both required and optional)
    • an SMTK attribute definition that describes information returned by the operator.
  • smtkCreateMacro: a macro to provide a static method for creating a shared pointer to a new instance
  • smtkDeclarModelOperator: declar methods for accessing the operator’1 class-name and human-readable name.
  • smtkImplementsModelOperator: register the operator with the proper session. It ties C++ class name, attribute definition and registeration
    + that's how you register operators, sessions with `smtkComponentInitMacro`

  • macro to encoding an XML file as a variable in a C++ header. #include "implement_an_operator_xml.h"

  • operator::log(): use this for errors and informational messages so that the user has a chance to see them

New operator manager

  • A lot of this code is still in flux, but the issue we are trying to tackle is that operators are currently only held by a particular session, so there are much fewer operations that may collide by having the same name. Now that we are moving to a global OperatorManager, all of these operators must live in the same place. An attribute system uniquely identifies its attributes by name, so now each operator must have a unique name in the attribute system.

The first step to resolving this issue is to store operators by their fully qualified class names (namespace + name). Fortunately, std::type_index provides us with a unique mapping between a class and a value with no additional overhead, so we are leveraging this to construct a map to operator constructors.

In this MR, the interaction between smtk::model::Operators and smtk::model::Sessions is unchanged from its original design. I have simply introduced an additional means of constructing operators via the operator class’s type index, as an incremental change to the design. Once this lands, the next step will be to modify smtk::model::Session to use the operator manager to construct operators, manager operator instances and possibly rebrand operators and their attributes (effectively renaming user-facing parts of the operator definition to match the context of the session).

nlohmann_json

  • json::parse(): treat it as a json instead of as a string, which means deserialize a string into json. Appending “_json” also works for deserialization
  • json::dump(): serialization
  • It can also be treated as a STL container
  • Arbitrary types conversions
    • provide to_json and from_json function
    • Those methods must be in your type’s namespace
    • type must be default constructible
    • use at() to access the object values rather than operator[]
    • If several operator= is defined, use decltype
    • Be careful with the definition order of the from_json/to_json functions: If a type B has a member of type A, you MUST define to_json(A) before to_json(B)
  • adl_serializer: Argument-dependent lookup, used for coverting thid-party types
  • if the type is not default constructible and you want to use the get() method
