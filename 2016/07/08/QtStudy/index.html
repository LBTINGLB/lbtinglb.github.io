Parenting system

  • When an object is destroyed, all of its children are destroyed as well. So, calling delete becomes optional in certain cases.
  • All QObjects have findChild and findChildren methods that can be used to search for children of a given object.
  • Child widgets in a QWidget automatically appear inside the parent widget.

model/view framework

  • Model/View Tutorial
    • Selection + different views
  • model/view architecture and delegate
  • Models
    • All item models are based on the QAbstractItemModel class.
  • Views
    • All views are based on the QAbstractItemVIew class
    • how to set up a view to display the items in the model: calling its setModel() function
  • Delegates
    • All delegates are based on the QAbstractItemDelegate class
  • Sorting
    • If the model is sortable, just reimplement QAbstractItemModel::sort() function
    • Or you can enable interactive sorting. Check the link in title
  • Model classes
    • model index: each piece of information that can be obtained via a model is represented by a model index(If a long-term reference is needed, use QPersistentModelIndex)
    • If an invalid model index is specified for the parent item when an index is requested using index(), the index returned refers to a top-level item in the model.
  • View classes
    • view can be constructed without a model, but a model must be provided before it can display useful information.
    • item roles: stored on model. Qvariant value = model->data(index, role);
    • existing view: QListView, QTreeView and QTableView(aka spreadsheet)
    • Handling selections of items: QItemSelectionModel
    • sharing selections among views: secondTableView->setSelectionModel(firstTableView->selectionModel());
  • Delegate classes
    • Unlike the Model-View-Controller pattern, the model/view design does not include a completely separate component for managing interaction with the user. So delegates are expected to be able to render their contents themselves by implementing the paint() and sizeHint() functions.
    • provide an editor, and the delegate must provide a function to copy model data into the editor.
    • functions must be implemented: createEditor(), updateEditorGeometry,set EditorData() and setModelData()
  • Handling selections in item views
    • QItemSelectionModel: mainntains model indexes for items in a single model, and is independent of any views.
    • Selections are applied to a collection of model indexes held by a selection model
    • two important signals - selectionChanged and currentChanged
    • updating a selection:
      • “QItemSelectionModel::Toggle” flag: The Toggle flag causes the selection model to invert the state of the specified items, selecting any deselected items given, and deselecting any currently selected items.
  • Creating new models
    • Designing a model
      • Make the model editable: implement flags()(make the delegate knows that items are editable) and setData()(provide a way for the delegate to set the data in the model) functions
      • When the data has been set, the model must let the views know that some data has changed. This is done by emitting the dataChanged() signal
      • In order to customize the headerData, you should overwrite headerData function in model

QAbstractProxyModel

-tutorial

  • QAbstractProxyModel class: sorting, filtering or other data processing tasks whose sole role is to allow the data to be manipulated without affecting the source data model; this way the source data model can be used by any number of different proxy modes and each proxy model can manipulate the data in different ways without effecting each other.
  • subclass: QIdentityProxyModel: proxies its source model unmodified
    • can be used to forward the structure of a source model exactly, with no sorting, filtering or other transformation. This is similar in concept to an identity matrix where A.I = A.
  • sublcass: QSortFilterProxyModel:support for sorting and filtering data passed between another model and a view
    • can be used for sorting items, filtering out items, or both. The model transforms the structure of a source model by mapping the model indexes it supplies to new indexes, corresponding to different locations, for views to use. This approach allows a given source model to be restructured as far as views are concerned without requiring any transformations on the underlying data, and without duplicating the data in memory.
    • By default, the model does not dynamically re-sort and re-filter data whenever the original model changes. This behavior can be changed by setting the dynamicSortFilter property.
