Learning notes

  • show the current path: pwd
  • show the hidden files: ls -ah
  • [ ] initialize a git repository
    • git init
  • [ ] add a file to git repository
    • 1st step: git add “filename”
      • add many files of the same type: git add '*.txt'
    • 2nd step: git commit -m “comment”
      • -m means that you are writing the explanation for current push
  • [ ] check status and see difference
    • git status # tell you about the current status of the repository
    • git diff # find out what has changed
    • git diff HEAD -- <file name>
    • git diff HEAD
      • we want the diff of our most recent commit, which we can refer to using the HEAD pointer.
    • git diff --staged
      • check the changes we just staged
  • [ ] check version and go back
    • git log
    • git log --pretty=oneline # show files in one linegit
    • git reset -- hard HEAD^
      • git reset --hard branchName
      • HEAD^ is the last version, HEAD^ is last and last version, you can use HEAD~100 to go to 100 versions back
    • git reflog
      • a writeup for all your records

  • have conflict after git rebase
    • Ex. you branch is x -> y, and remote is x -> z. you need to tell git which direction you want to go. So, type git add y then git would use your change rather than the change on the remote.
  • merge check fails on gitlab
    • After fix the problem, use Do: check to check merge availability again.
  • git reset --hard <commit>:
    • it’s easy for you to have some files that have conflict(or just exist as the master). SO you would see Please move or remove them before you can switch branches. Use git clean -dfn first to check the files you want to clean, then use git clean -df to actually clean the folder.

show how many files have been modified in a change or compare with other commit

  • git show --stat
  • git whatchanged -n, n is the number of commits
  • git diff <SHA1-of-from-commit>..<SHA1-of-to-commit> --name-only

find out which commit causes a failure or bug - git bisect

  • git bisect start/good/bad, you would love it!
  • git bisect run make/ninja tests/your-batch.sh

create several commits out of current unstaged changes

  • git add -p

release and master

  • In order to merge a patch to release and master:
    • Checkout release branch, apply your patch -> releaseWithPatch
    • Checkout master, rebase on releaseWithPatch
    • Create a MR
  • Do this to SMTK, then same to CMB. But: 1. cmb release branch + patch -> bump SMTK SHA -> rebase on master
