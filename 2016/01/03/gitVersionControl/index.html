Learning notes

  • show the current path: pwd
  • show the hidden files: ls -ah
  • [ ] initialize a git repository
    • git init
  • [ ] add a file to git repository
    • 1st step: git add “filename”
      • add many files of the same type: git add '*.txt'
    • 2nd step: git commit -m “comment”
      • -m means that you are writing the explanation for current push
  • [ ] check status and see difference
    • git status # tell you about the current status of the repository
    • git diff # find out what has changed
    • git diff HEAD -- <file name>
    • git diff HEAD
      • we want the diff of our most recent commit, which we can refer to using the HEAD pointer.
    • git diff --staged
      • check the changes we just staged
  • [ ] check version and go back
    • git log
    • git log --pretty=oneline # show files in one linegit
    • git reset -- hard HEAD^
      • HEAD^ is the last version, HEAD^ is last and last version, you can use HEAD~100 to go to 100 versions back
    • git reflog
      • a writeup for all your records
  • [ ] learning directory & repository & stage
    • learning directory -> learnGit folder
    • repository -> .git ban
    • stage
  • [ ] How to cancel the changes
    • [] resetting the Stage
      • git reset <file>
        • we unstage the files (but the file is still there.)
    • [ ] discard changes in the working directory(You want to discard changes, and they have not already been added.)
      • git checkout -- <file>
        • – is important because without –, checkout would become switch to another branch
    • [ ] discard changes in the repository (You want to discard changes, and they have already been added.)
      • git reset HEAD <file>
        • by using git reset, you can retreat the change from reprository to working area
      • then use git checkout -- <file>
    • [ ] already been added and commited
      • use git reset – hard HEAD
  • [ ] delete files
    • two options
      • you really want to delete the file from git repository
        • git rm <filename>
          • it would remove the actual files from the disk and stage the removal of the files
        • git commit -m “Add the comment you want”
      • you want to retrive the file back
        • git checkout –
  • [ ] delete a branch
    • git branch -d <branch name>
  • [ ] push the local files to the remote repository (such as github)
    • connect a remote repository
      • git remote add origin git@server-name:path/repo-name.git
    • Pushing Remotely
      • git push -u origin master
        • origin: the name of our remote
        • master*: the name of the default local branch
        • -u: it tells Git to remember the parameters, so that next time we can simply run git push and git will know what to do
    • git push origin master
      • if you want to push the newest update, just use git push origin master
  • [ ] check for changes on our Github repository sand pull down any new changes
    • git pull origin master
  • [ ] how to clone repository from remote
  • [ ] build and combine branch
    • why create a new branch? - you can keep your own work in your own respository while not interfering other people
    • git checkout -b <new branch name>
      • create a new branch
      • it works as two commands 1. git barnch dev 2. git checkout dev
    • git branch
      • check current branch
    • git merge <the name of the branch you want to merge>
      • first, checkout to the branch that you want as the final one.
      • type git merge <branch with changes>
    • git branch -d <branch name>
      • delete the branch
  • [ ] merge confliction
    • git log --graph --pretty=oneline --abbrev-commit
    • git log --oneline --graph --all --decorate
      • show the git log in a beautiful graph
    • when there is an conflict in git so that it could not merge by itself, you need to solve the conflict first.
  • [ ] how to manage confliction and management strategy
    • `git merge –no-ff -m ““ dev
      • do not merge with Fast forward model
      • Using FF model cannot let you view the merge history, but normal forward model can.
    • keep master stable, work on other repository
  • [ ] bug branch
    • if you have a bug, you can use a temporary branch to fix it. After the repairment, you can merge them and delete the temporary branch.
    • git stash
      • Stash the changes in a dirty working directory away
        git
    • git stash apply
      • restore the working directory but keep the stash content(you need to insert git stash drop to delete the stash)
    • git stash pop
      • restor the working directory and delete the stash content
    • summary: dev上开发时，需要修复master上的bug，使用git stash 保存dev现场后切换至master修复bug，修复完成后，为了保证dev上的相同bug被修复，需要将master合并至dev上：checkout dev切换回dev分支，git merge master合并master至dev上，则dev上已经修复的bug会被同步至dev上。
  • [ ] feature branch
    • when adding some new functionality, never did you want to distrub the main branch. So when adding a new feature, you’d better add a new feature branch. Work on it, after the whole development the merge and delete i.t
    • ‘git branch -D feature-vulcan’
      • delete a branch that is not fully merged
  • [ ] multiple people work on the same project
    • git push origin <branch name>
      • push local files to the github
        • multiple people working together guidance
  • [ ] build a tag for your repository
    • Sometimes you want to find a specific version. Instead of trying to find the commit id as “892b892a3f7”, you can find it via label. (git show v1.1)
    • git tag <tag name>
      • tag the current branch
      • git tag -a <tag name> -m <comment for the tag> <commit id>
        • tag a history version
      • git tag -d <tag name>
        • delete a tag
      • git tag
        • show the tag for the current branch
      • git push origin <tag name>
        • push tag to the remote repository
      • git push origin tags
        • push tags to the remote repository
      • git push origin :refs/tags/<tagname>
        • delete a tag on a remote repository

    • it means the files we have told git that are ready to be committed.
  • if you are working on a bug branch, and the origin/master changed:
    • do git merge origin/master with the bug branch
    • solve the conflicts. Then your bug branch would be in update with the orgin/master branch

    • get fetch –all

    • the mean of this command is that now you push your local new branch to your gitlab account. In the future, you can just safely type git push to push all the changes.

  • How to Write a Git Commit Message
      1. Separate subject from body with a blank line
        • Example: “Fix typo in introduction to user guide”
    1. Limit the subject line to 50 characters
    1. Capitalize the subject line
    1. Don not end the subject wline with a period
    1. Use the imperative mood in the subject line
      • spoken or written as if giving a command or instruction
    1. Wrap the body at 72 characters
    1. Use the body to explain what and why vs. how

  • git commit -amend

    • Print lines matching a pattern
