Preface

  • design patterns + generic programming + C++
  • horizontal: a small amount of library code implements a combinatorial- and essentially openeded-number of structures and behaviors
  • vertical: applicable to a vast range of programs
  • Generic programming: a paradigm that focuses on abstracting types to a narrow collection of functional requirements and on implementing algorithms in terms of these requirements.

    1.5 Policies and Policy Classes

  • policy class is not intended for stand-alone use; instead, they are inherited by, or contained within, other classes
  • The classes that use one or more policies are called hosts or host classes.
  • policy-based class design

    1.6 Eneriched Policies

  • The resulting context is very favorable. Clients who need enriched policies can benefit from that rich
    functionality, without affecting the basic functionality of the host class. Don’t forget that users—and not
    the library—decide which policy class to use. Unlike regular multiple interfaces, policies give the user the
    ability to add functionality to a host class, in a typesafe manner.

    1.7 Destructors of Poliyc Classes

  • Virtual destructor should be avoided since it incurs some size overhead for the objects of that class and the calss don’t have any data members.
  • A nonvirtual protected destructor should be used.

    1.8 Optional Functionality Through Incomplete Instantiation

  • If a member function of a class template is never used, it’s not even instantiated
  • So the manager would benefit from optional enriched interfaces and work correctly with poorer inferface

Jargons

  • boilerplate code
    • boilerplate code or boilerplate refers to sections of code that have to be included in many places with little or no alteration.

      Visitor design pattern

  • SOURCE MAKING example
  • Benefit:
    • In essence, the visitor allows adding new virtual functions to a family of classes, without modifying the classes.
    • Visitor’s primary purpose is to abstract functionality that can be applied to an aggregate hierarchy of “element” objects. The approach encourages designing lightweight Element classes - because processing functionality is removed from their list of responsibilities. New functionality can easily be added to the original inheritance hierarchy by creating a new Visitor subclass.

Decorator Design pattern

  • Behaviors are allowed to be added to an individual object, either statically or dynamically without affecting the behavior of other objects from the same class
  • Objective: responsibilities should be added to(and remove from) an object dynamically at run-time
  • vs subclassing: Subclassing adds behaviors at compile time, and the change affects all instances of the original classes. Decorating can provide new behavior at run-time for selected objects
  • when to use: If we cannot predict, at design time, what combinations of extensions will be needed. Which means that a new classes would be made for every possible combination
  • c++ implementation

Curiously recurring template pattern

  • An idiom in C++ where a class X derives from a class template instantiation using X itself as template argument. More generally it’s known as F-bound polymorphism. It’s used for static polymorphism
  • You can use members of the derived class within its own member functions
  • You can achieve a similiar effect to the use of virtual functions, without the cost(and some flexibility) of dynamic polymorphism. The overridden functions in the derived class will be selected at compile time. This effectively emulates the virtual function call system at compile time without the costs in size or function call overhead (VTBL structures, and method lookups, multiple-inheritance VTBL machinery) at the disadvantage of not being able to make this choice at runtime.

Structural patterns in a simple way

Creational design patterns

Abstract Factory

  • Brief introduction: Provide an interface for creating families of related or dependent objects without specifying their concrete classes.
  • Problem to solve: Making the application portable can be hard. Two many conditions to support.
  • Benefits: This mechanism makes exchanging product families easy because the specific class of the factory object appears only once in the application - where it is instantiated. The application can wholesale replace the entire family of products simply by instantiating a different concrete instance of the abstract factory.
  • Check list:
    • Decide if “plateform independence” and creation services are the current source of pain
      • Define a factory derived class for each platform that encapsulates all references to the new operator.

Builder Design Pattern

  • Brief introduction: Separate the construction of a complex object from its representation so that the same construction process can create different representations. A director invokes the builder services as it interprets the external format. Unlike creational patterns that construct products in one shot, the Builder pattern constructs the product step by step under the control of the director.
  • Check list:
    • Decide if a common input and many possible representations(or outputs) is the problem at hand.
  • Example: Ordering foor in MC. Build burger? donouts? fries?

Factory Method

  • Brief introduction: Define an interface for creating an object, but let the subclasses decide which class to instantiate.
  • Problem to solve: A framework needs to standardize the architectural model for a range of applications, but allow for individual applications to define their own domain objects and provide for their instantiation.
  • Factory Method is similar to Abstract Factory but without the emphasis on families.

Object Pool Design Pattern

  • Brief introduction: Used in situations where the cost of initializing a class instance is hight, the rate is high but the number of instantiations in use at any one time is low.
  • Problem to solve: Object pools (otherwise known as resource pools) are used to manage the object caching. A client with access to a Object pool can avoid creating a new Objects by simply asking the pool for one that has already been instantiated instead. Generally the pool will be a growing pool, i.e. the pool itself will create new objects if the pool is empty, or we can have a pool, which restricts the number of objects created.
  • Think it as a warehouse. It gives out tools when there are unused ones and reclaim then they are not in use.

Prototype Design Pattern

  • Brief introduction: Specify the kinds of objects to create using a prototypical instance, and create new objects by copying this prototype.
  • Problem to solve: Application hard wires the class of object to create in each “new” expression

Singleton Design Pattern

  • Brief introduction: Ensure a class has only one instance, and provide a global point of access to it.
  • Problem to solve: Application needs one, and only one, instance of an object. Additionally, lazy inititialization and global access are necessary.
  • Ownership of the single instance cannot be reasonably assigned + lazy initialization is desirable + Global access

Structural patterns

Structural design patterns

Adapter - Match interfaces of different classes

  • brief introduction: It’s used to convert the interface of a class into another interface which clients expect(Wrap an existing class with a new interface). It creates an intermediary abstraction that translates, or maps, the old componment to the new system.
  • Problem to solve: An “off the shelf” component offers compelling functionality that you would like to reuse, but its “view of the world” is not compatible with the philosophy and architecture of the system currently being developed.
  • adapter vs proxy vs decorateor vs facade: different interface vs same interface vs enhanced interface vs new

Bridge Design Pattern - Separates an object’s interface from its implementation

  • brief introduction: Decouple an abstraction from its implementation so that two can vary independently. Interface in an inheritance hierarchy + implementation in its own inheritance hierarchy(Decouple the problem into orthogonal classes hierarchies). It uses handle/body idiom
    • The interface object is known to the client while the implementation object, or “body”, is safely encapsualted to ensure that it may continue to evolove, or be entirely replaced
  • Problem to solve: “Hardening of the software arteries” has occurred by using subclassing of an abstract base class to provide alternative implementations. This locks in compile-time binding between interface and implementation. The abstraction and implementation cannot be independently extended or composed. The number of classes you need is exponential.
  • Benefit:
    • run-time bindng of the implementation
    • a proliferation of classes resulting from a coupled interface and numerous implementations
    • share an implementation among multiple objects
    • hide the detail from the client
    • improve the extensibility.
  • Check list:
    • decide if two orthogonal dimensions exist in the domain. abstraction/plateform, domain/infrastructure, or front-end/back-end, or interface/implementation
    • Design the separation of concerns: what does the client want, and waht do the platforms provide.
    • create the abstraction base class that “has a” platform object and delegates the platform-oriented functionality to i
  • adapter vs bridge
    • Adapter make things work after they’re designed; Bridge makes them work before they are.

Composite - A tree structure of simple and composite objects

  • brief introduction: treat individual objects and compositions of objects uniformly.
  • Problem to solve: Application needs to manipulate a hierarchical collection of “primitive” and “composite” objects. Processing of a primitive object is handled one way, and processing of a composite object is handled differently. Having to query the “type” of each object before attempting to process it is not desirable.
  • Read Gang of Four for a discussion of `safety versus transparency trade-offs.
  • benefit:
    • the composite can be treated atomically, just like a leaf.

Decorator - Add responsibilities to objects dynamically

  • brief introduction: attach additional responsibilities to an object dynamically - an alternative to subclassing for extending functionality
  • Check list:
    • a single core(or non-optional) component, several optional embellishments or wrappers, and an interface that’s common to all
    • create a Lowest common denominator(LCD) interface that makes all classes interchangeable
    • the core and decorator classes inherit from the LCD interface. The Decorator class declares a composition relationship to the LCD interface, and this data member is initialized in its constructor.

Facade - A single class that represents an entire subsystem

  • brief introduction: Provide a unified interface to a set of interfaces in a subsystem. It defines a higher-level interface that makes the subsystem easier to use.
  • Problem to solve: A segment of the client community needs a simplified interface to the overall functionality of a complex subsystem.
    • so the facade object should be a fairly simple advocate or facilitator. It should not become an all-knowing oracle or ‘god’ object.
  • check list:
    • identify a simpler, unified interface for the subsystem or component
  • benefit:
    • It can make multiple interfaces work together

Flyweight - A fine-grained instance used for efficient sharing

  • brief introduction: Use sharing to support large numbers of fine-grained objects efficiently
  • Problem to solve: Designing objects down to the lowest levels of system “granularity” provides optimal flexibility, but can be unacceptably expensive in terms of performance and memory usage.
  • check list:
    • ensure that object overhead is an issue needing attention. And the client is willing to absorb the responsibility realignment
    • divide the target class’s state into: shareable(intrinsic) state and non-shareable(extrinsic)state
    • Create a Factory that can cache and reuse existing class instances.

Private Class Data - Restricts accessor/mutator access

  • brief introduction: Control write access to class attributes. The motivation for this design pattern comes from the design goal of protecting class state by minimizing the visibility of its attributes (data).
  • example: vtkInternal ``` * `

Proxy - An object representing another object

  • brief introduction: Provide a surrogate or placeholder for another object to control access to it. Use an extra level of indirection to support distributed, controlled, or intelligent access`
  • Add a wrapper and delegation to protect the real component from undue complexity
  • Problem: You need to support resource-hungry objects, and you do not want to instantiate such objects unless and until they are actually requested by the client.(vtkPipeLine!)
  • four common situations in which the proxy pattern is applicable
    • A virtual proxy is a placeholder for “expensive to create” objects. The real object is only created when a client first requests/accesses the object.
    • A remote proxy provides a local representative for an object that resides in a different address space. This is what the “stub” code in RPC and CORBA provides.
    • A protective proxy controls access to a sensitive master object. The “surrogate” object checks that the caller has the access permissions required prior to forwarding the request.
    • A smart proxy interposes additional actions when an object is accessed. (reference coutning, lock checking)

Behavioral design patterns

Chain of Responsibility

  • brief introduction: avoid coupling the sender of a request to its receiver by giving more than one object a chance to handle the request
  • problem to solve: Need to efficiently process the requests without hard-wiring handler relationships and precedence
  • Now each sender keeps a single reference to the head of the chain, and each receiver keeps a single reference to its immediate successor in the chain. Do not use it when each request is only handled by one handler
  • check list:
    • the base class matains a next pointer.
    • each derived class implements its contribution for handling the request.

Command Design Pattern

  • brief introduction: Encapsulate a request as an object, promoting “invocation of a method on an object”
  • example: check at a dinner

Interpreter Design Pattern

  • Map a domain to a language, the language to a grammar, and the grammar to a hierarchical object-oriented design.

Mediator Design Pattern

  • brief introduction: Define an object that encapsulates how a set of objects interact. Mediator promotes loose coupling by keeping objects from referring to each other explicitly, and it lets you vary their interaction independently.
  • problem to solve: We want to design reusable components, but dependencies between the potentially reusable pieces demonstrates the “spaghetti code” phenomenon (trying to scoop a single serving results in an “all or nothing clump”). (Permission to access a system)

Memento Design Pattern

  • problem to solve: Need to restore an object back to its previous state (e.g. “undo” or “rollback” operations).
  • three distinct roles:
    • Originator - the object that knows how to save itself
    • Caretaker - the object that knows why and when the Originator needs to save and restore itself
    • Memento - the lock box that is written and read by the Originator, and shepherded by the Caretaker.

Null Object Design Pattern

  • brief introduction: encapsulate the absence of an object by providing a substitutable alternative that offers suitable default do nothing behavior
  • problem to solve: Given that an object reference may be optionally null, and that the result of a null check is to do nothing or use some default value, how can the absence of an object — the presence of a null reference — be treated transparently?

Observer Design Pattern

  • brief introduction: Define a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.
  • Observers register themselves with the subject as they are created. When the subject changes, it broadcast to all registered observers that it’s changed.
  • Check list:
    • Differentiate between the core(independent) functionality - goes into the subject and the optional(dependent) functionality - goes into the observer

State Design Pattern

  • brief introduction: Allow one object to alter its behavior when its internal state changes. The object will appear to change its class.

Strategy Design Pattern

  • brief introduction: Define a family of algorithms, encapsulate each one, and make them interchangeable. Strategy lets the algorithm vary independently from the clients that use it.:w

Template Method Design Pattern

  • brief introduction: Define the skeleton of an algorithm in an operation, deferring some steps to client subclasses. Template Method lets subclasses redefine certain steps of an algorithm without changing the algorithm’s structure.

Visitor Design Pattern

  • brief introduction: define a new operation without changing the classes of the elements on which it operates
  • problem to solve: Many distinct and unrelated operations need to be performed on node objects in a heterogeneous aggregate structure. You want to avoid “polluting” the node classes with these operations.
  • discussion:
    • single dispatch: the operation executed depends on: the name of the request, and the type of receiver
      • double dispatch: the operation executed depends on: the name of the request, and the type of TWO receivers (the type of the Visitor and the type of the element it visits)
