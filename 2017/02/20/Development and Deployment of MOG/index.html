Server-Side MMO Architecture. Naïve, Web-Based, and Classical Deployment Architectures

Chapter VI(a). Server-Side MMO Architecture. Naïve, Web-Based, and Classical Deployment Architectures

  • naive architecture single flow design:
    • separate players coming from separate databases cannot interact with each other
    • hard to merge DB. two user with same ID on different server
  • Web-Based game Development Architecture
    • slow paced game, low interaction
    • front end + back end cache:
      • Front-end caching is your usual page caching (like nginx in reverse-proxy mode, or even a CDN)
      • back-end cache as a cache to reduce number of DB reads, but also will want to make it a write-back cache to reduce the number of DB writes
      • write-back cache
        • For example, if we’d need to save each and every click on the simulated farm with 25M daily users (each coming twice a day and doing 50 modifying-farm-state clicks each time in a 5-minute session), we could easily end up with 2.5 billion DB transactions/day (which is infeasible, or at least non-affordable). On the other hand, if we’re keeping write-back cache, we can write the cache into DB only once per 10 minutes, we’d reduce the number of DB transactions 50-fold, bringing it to much more manageable 50 million/day.
          • optimistic locking(ignoring possibility that some other Web Server also got the same “game world” and is working on it + Web Server is NOT allowed to send any kind of reply back to user yet + it works fine as long as collision probability ), pessimistic locking(multi-threaded mutex-based locking), transsactions(Redis transaction is merely a sequence of operations which are executed atomically. It means no locking, and an ability to split your “game world” state into several parts to deal with traffic)
          • merit
      • easy-to-find developers
      • simplicity and being relatively obvious (that is, until you need to deal with locks, see above)
      • web servers are stateless (except for caching, see below), so failure analysis is trivial: if one of your web servers goes down, it can be simply replaced
      • can be easily used both for the games with downloadable client and for browser-based ones
        • downsides
        • no way out of web-based architecture
        • it is pretty much HTTP-only (with an option to use Websockets); migration to plain TCP/UDP is generally not feasible.
        • as the number of interactions between players and game world grows, Web-Based Architecture becomes less and less efficient
        • you’ll need to spend A LOT of time meditating over your caching strategy.
  • Classical Game Deployment Architecture
    • Game Servers are traditionally divided according to their functionality
      ` + Game World Server + Matchmaking Server + Tournament Sever + Payment Server + Social Gat way Server
