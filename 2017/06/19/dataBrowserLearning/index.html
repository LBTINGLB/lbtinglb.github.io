JSON and XML comparison

  • JSON:
    • simple syntax with less markup
    • easy to use with JS
    • only a handful of different data types are supported
  • XML:
    • generalized markup, possible to create “dialects” for any kind of purpose
    • built in support of namespaces
    • relatively wordy compared to JSON

Models clarification

  • qtAbstractQuestModel
    • columnCount for each DSLModelItem
  • qtDSLModel(subclassing qtAbstractQuestModel)
    • handle entity creation & modification & deletion
  • DSLModelItem
    • internal base data structure of qtDSLModel
    • stores all data(name, created time, id and etc)
  • qtCollectionFeatureTableView
    • QSortFilterProxyModel based on qtDSLModel
  • qtCollectionDatasetTableView
    • qtDSLModel
    • qtCollectionFlatModel
      • subclassing QAbstractProxyModel, Internally maps the input tree beneath a Collection index to an array, with indexes ordered by their position in depth-first traversal
    • qtCollectionDatasetModel
      • suclassing QSortFilterProxyModel, displaying a table DSLDataset instances for a given collection

Design logic

  • qtGeoPreviewMap and qtGepProjectmap have DSLPointGeometry and DSLPolygonGeometry

how features are added in preview mode and normal mode

  • key function: Store::parseDSLFeature
  • Preview mode: Get a json from the server and pass it into qtGeospecificSearchPanel::onGetFeaturesResult -> Store::parseDSLFeature
  • normal mode(add features&datasets into a collection): qtGeospecificSearchPanel::onAddResultsClicked gets a list of feature IDs and target collection -> qtDataManager::addDSLFeatures which would add features to dsl(Given a list of feature uri, make rpc calls to get the json)&update datastore(Store::parseDSLFeatures)&update dslModel

Point vs Line vs Polygon

  • point:
    • coordinate format: [x, y]
  • line:
    • coordinate format: [[[x, y], [x, y]], [[x,y],[x,y]]]
  • polygon:
    • coordinate format: [[[x, y], [x, y]]]
